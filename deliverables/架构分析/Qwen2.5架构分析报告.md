# Qwen2.5 架构分析报告(对比 Llama3.2)

> 数据来源:本地 `count_params.py` 统计 + `config.json` 字段,真实跑出。
> 对比模型:`Qwen2.5-1.5B-Instruct` vs `Llama-3.2-3B-Instruct`。

---

## 一、参数量统计(count_params.py 输出)

**Qwen2.5-1.5B** — 总计 1,543,714,304(1.544B)

| 模块 | 参数量 | 占比 |
|---|---|---|
| mlp 前馈 | 1,156,055,040 | **74.9%** |
| embedding 词表 | 233,373,696 | 15.1% |
| attention 注意力 | 154,198,016 | 10.0% |
| norm 归一化 | 87,552 | 0.0% |

**Llama-3.2-3B** — 总计 3,212,749,824(3.213B)

| 模块 | 参数量 | 占比 |
|---|---|---|
| mlp 前馈 | 2,113,929,216 | **65.8%** |
| attention 注意力 | 704,643,072 | 21.9% |
| embedding 词表 | 394,002,432 | 12.3% |
| norm 归一化 | 175,104 | 0.0% |

**观察**:
- **两者都是 MLP 占大头**(65–75%)。前馈层是参数最密集的地方(SwiGLU 三个大矩阵)。
- **Qwen 的 embedding 占比更高(15.1% vs 12.3%)**:因为 Qwen 词表巨大(15.2 万 vs 12.8 万),而模型本体更小,词表相对更"重"。
- **Llama 的 attention 占比更高(21.9% vs 10.0%)**:因为 Llama GQA 分组更保守(8 套 KV vs Qwen 2 套),KV 投影矩阵更大;加上 hidden_size 翻倍。

## 二、config.json 字段逐项对比

| 字段 | Qwen2.5-1.5B | Llama-3.2-3B | 含义 |
|---|---|---|---|
| hidden_size | 1536 | 3072 | 隐藏维度(模型"宽度") |
| num_hidden_layers | 28 | 28 | 层数(模型"深度")—— 巧合相同 |
| num_attention_heads | 12 | 24 | Query 头数 |
| num_key_value_heads | **2** | **8** | KV 头数 → **两者都用 GQA** |
| GQA 分组比 | **6:1** | 3:1 | 每几个 Q 共享一套 KV |
| intermediate_size | 8960 | 8192 | MLP 中间层维度 |
| vocab_size | **151936** | 128256 | 词表大小(Qwen 更大,对中文友好) |
| rope_theta | **1000000** | 500000 | RoPE 基础频率 |
| max_position_embeddings | 32768 | **131072** | 最大上下文(Llama 达 128K) |
| tie_word_embeddings | True | True | 输入/输出词表共享权重 |
| hidden_act | silu | silu | 激活函数(配合门控 = SwiGLU) |

**相同点(现代 LLM 的共识设计)**:GQA、RoPE、SwiGLU(silu 门控)、RMSNorm、tie embedding —— Qwen 和 Llama 在骨架上高度趋同。

**差异点**:
1. **词表**:Qwen 15.2万 > Llama 12.8万。Qwen 为中文塞了更多词,单位中文更省 token。
2. **GQA 力度**:Qwen 6:1 比 Llama 3:1 更激进 → KV cache 更省,推理更轻。
3. **上下文**:Llama 3.2 的 max_position 达 **131072(128K)**,远超 Qwen 的 32768,但 rope_theta 反而更小(靠 RoPE scaling 外推)。

## 三、GQA(分组查询注意力)深入

### 三种注意力对比
| 方式 | Q 头 | KV 头 | 特点 |
|---|---|---|---|
| MHA 标准多头 | N | N | 质量最好,KV cache 最大 |
| MQA 多查询 | N | 1 | 最省,但掉质量 |
| **GQA 分组** | N | 少数几组 | 折中(Qwen/Llama 都用) |

### 为什么用 GQA
推理时每生成一个新 token,都要回看前面所有 token 的 **K、V**(存显存,叫 **KV cache**),这是长文本推理的**显存瓶颈**。减少 KV 头数 = 缩小 KV cache:
- Qwen(12 Q / 2 KV):KV cache 比全 MHA **小 6 倍**
- Llama(24 Q / 8 KV):小 3 倍

Q 头数不变 → 保持表达多样性 → 质量几乎不降。这也解释了为什么 **Qwen 的 attention 参数只占 10%**(K/V 投影矩阵被压小了)。

### 口头解释版
> GQA 让多个 Query 头共享少数几套 Key/Value 头(Qwen 是 12 共享 2)。目的是缩小推理 KV cache、省显存、加速长文本,而质量几乎不降,是 MHA 与 MQA 的折中。

## 四、RoPE(旋转位置编码)深入

### 解决什么问题
注意力本身**没有顺序感**("猫追狗"和"狗追猫"看起来一样),必须额外注入位置信息。

### 怎么做
RoPE **按 token 位置把 Q、K 向量旋转一个角度**,位置越靠后转得越多。旋转后两向量点积**只依赖相对距离 (m−n)** → 天然编码**相对位置**。且 **RoPE 无可训练参数**(所以 count_params 里看不到它)。

### rope_theta 的作用
把向量各维度想象成一组转速不同的钟表指针,`rope_theta` 控制整体转速:
- **theta 越大 → 转得越慢 → 可编码的位置范围越长**(避免远处位置"绕圈撞车")。
- Qwen2.5 = 1,000,000;Llama3.2 = 500,000;初代 Llama 仅 10,000 → 越新越大,越擅长长文本。

### 口头解释版
> RoPE 通过按位置旋转 Q、K 向量注入位置信息,点积只依赖相对距离,编码的是相对位置;rope_theta 是旋转基础频率,越大越能外推长文本。它不含可训练参数。

## 五、推理行为对比(inference.py 实测)

同样 3 个 prompt,两个模型表现:

| 领域 | Qwen2.5-1.5B | Llama-3.2-3B |
|---|---|---|
| 代码生成 | 用 `isalnum()` 正确处理标点,回文判断**逻辑正确** | 只 `replace(" ","")` 删空格,**注释声称含标点例子返回 True,实际会返回 False(有 bug)** |
| 逻辑推理 | 正确(小明 15 岁) | 正确(小明 15 岁) |
| 角色扮演 | 能扮物理老师,例子略牵强 | 能扮,但夹了英文单词 "Imagine"(中英混杂瑕疵) |

**观察**:
- Llama-3.2-3B 参数量是 Qwen1.5B 的 2 倍,但**中文任务上并没有明显更强**——中文代码注释、角色扮演反而出现小瑕疵。这与 **Qwen 词表更偏中文、中文语料更多**有关。
- 小模型(1.5–3B)在**复杂逻辑仍可靠**(年龄推理都对),但在**代码正确性、语言纯度**上会露怯 → 为后续"微调 / 更大模型"提供动机。

---

## 六、结论

1. **骨架趋同**:Qwen2.5 和 Llama3.2 都采用 GQA + RoPE + SwiGLU + RMSNorm + tie embedding,是当代 LLM 的主流范式。
2. **Qwen 的取舍**:更大词表(偏中文)+ 更激进 GQA(6:1,省显存)+ 更大 rope_theta,定位"中文友好 + 高效推理"。
3. **Llama 的取舍**:更宽的 hidden + 更保守 GQA + 超长 128K 上下文,定位"长文本 + 英文生态"。
4. **参数都堆在 MLP**;小模型里词表 embedding 占比可观(Qwen 尤甚)。
